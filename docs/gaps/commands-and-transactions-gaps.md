# Gaps: Commands and Transactions

Thematic gap analysis of rntme's command path (graph-IR `emit`, event-store append, Kafka relay, projection-consumer apply) vs. Medusa.js workflow + event-bus architecture, scoped to what a commerce-class service (Cart → Order → LineItem → Payment) would demand from the platform. Input evidence: rntme `graph-ir-compiler` command-runtime, `event-store` + relay, `projection-consumer`, the `issue-tracker-api` demo composite command graph, and Medusa survey B (`docs/superpowers/reports/2026-04-14-medusa-survey-b-module-workflow.md`).

Scope reminder — rntme is a **per-service** artifact runtime inside a larger LLM-agent-driven DDD platform. **Cross-service** workflows (Cart→Payment→Inventory→Shipping sagas) belong to **Zeebe**, not rntme. The only transaction-shape gap that survives for rntme is **intra-service multi-aggregate commands** (spec §8, question 1). The "Intersections with out-of-scope" section below records the Zeebe / ksqlDB / scheduled-jobs boundaries so readers start with them clear.

## What rntme has today

- **Command role** is inferred from a graph with exactly one `emit` node, compiled by `compileCommand()`. The compiler hard-rejects multi-aggregate by returning `CMD_MULTI_AGGREGATE_NOT_ALLOWED` — `packages/graph-ir-compiler/src/command-runtime/compile.ts:80`: "MVP: exactly one aggregate per command". One command = one aggregate stream, always.
- **Command execution** is a single-stream atomic append: `executeCommand()` takes `compiled.emits[0]!` as the head, builds one `stream = aggregate-aggregateId`, replays history, runs an OCC check against `expectedVersion`, and appends events in one SQLite transaction — `packages/graph-ir-compiler/src/command-runtime/execute.ts:61` and `packages/graph-ir-compiler/src/command-runtime/execute.ts:90`. Concurrency conflicts surface as `COMMAND_CONCURRENCY_CONFLICT`.
- **Composite commands** today are limited to **read-prelude guard + single emit**. The `issueTracker` demo's `assignIssueWithCapacityGuard.json` graph is the richest example — four read/filter/reduce nodes compute a capacity predicate, then one `emit` appends `IssueAssigned` (`demo/issue-tracker-api/src/artifacts/graphs/assignIssueWithCapacityGuard.json:47`). The read prelude runs against QSM before the write, so it is a pre-check, not a transactional read across two streams.
- **Event-store is SQLite**, one `event_log` table, one `publish_cursor` table. `appendEvents()` wraps the whole append in `db.transaction(...).immediate()` and enforces per-stream monotonic version via `SELECT COALESCE(MAX(version), 0)` followed by inserts — `packages/event-store/src/store/sqlite.ts:38` through `packages/event-store/src/store/sqlite.ts:96`. No cross-stream transactional API is exposed.
- **Relay to Kafka** is a polling outbox-by-table: the relay reads `event_log` rows after `publish_cursor`, produces to Kafka with retry+backoff, then advances the cursor — `packages/event-store/src/relay/loop.ts:35`. Delivery is at-least-once (a crash after send and before cursor write re-delivers). There is no explicit `outbox` table or "mark-as-sent" state; the relay conflates outbox and event log.
- **Projection consumer** is at-least-once + idempotent apply. `createProjectionConsumer` does `BEGIN IMMEDIATE → applyEvent* → COMMIT → commitOffsets` per Kafka batch — `packages/projection-consumer/src/consumer.ts:30`. `applyEvent` compares `last_event_version` per row and skips if `currentVersion >= envelope.version` (`packages/projection-consumer/src/apply/apply-event.ts:26`). This is spec §6.5 idempotency, not exactly-once.
- **No API-level idempotency key.** The demo Hono server routes commands via `createBindingsRouter` and passes request → `executeCommand` directly; there is no dedupe header plumbing and no request/result cache (`demo/issue-tracker-api/src/server.ts:45`).
- **No scheduled / cron primitive.** Commands only fire on inbound HTTP. Time-based triggers (reminder emails, session-expiry, abandoned-cart sweep) have no artifact today.

## How Medusa handles it

- **Workflows = durable DAGs of steps.** `createWorkflow(id, fn)` composes `createStep(name, invoke, compensate)` nodes; `TransactionOrchestrator` drives invoke-then-compensate-on-failure (`research/medusa/packages/core/workflows-sdk/src/utils/composer/create-step.ts:434`). Workflows are the unit of transactional guarantee, not the raw event bus.
- **Compensation is first-class.** Every step can declare a compensation function; failures walk back in reverse topological order (`research/medusa/packages/core/orchestration/src/transaction/transaction-orchestrator.ts:191`). `StepResponse.permanentFailure(msg, compensateInput)` distinguishes retryable vs. terminal failures (`research/medusa/packages/core/workflows-sdk/src/utils/composer/helpers/step-response.ts:15`).
- **Idempotency at workflow and step level.** `workflow.run({ input, idempotencyKey })` keys the checkpoint; `context.idempotencyKey` is available to every step; step results are cached in `TransactionContext.invoke` and re-used on replay (`research/medusa/packages/core/workflows-sdk/src/utils/composer/helpers/create-step-handler.ts:12`, `research/medusa/packages/core/orchestration/src/transaction/distributed-transaction.ts:85`). A workflow can mark `idempotent: false` when side effects are not idempotent (e.g. payment capture).
- **Durable checkpointing.** `TransactionCheckpoint` persists flow state via `IDistributedTransactionStorage`; the default in-memory impl is swappable for a persistent one, and on restart the orchestrator resumes from the last incomplete step (`research/medusa/packages/core/orchestration/src/transaction/distributed-transaction.ts:121`, `research/medusa/packages/core/orchestration/src/transaction/transaction-orchestrator.ts:100`).
- **Grouped events = transactional event emission.** Domain events are staged under an `eventGroupId` via `@EmitEvents()` and only released on success: `releaseGroupedEvents(eventGroupId)` fires them, `clearGroupedEvents(eventGroupId)` drops them on rollback (`research/medusa/packages/core/types/src/event-bus/event-bus-module.ts:66`). This is Medusa's version of a "transactional outbox" bolted onto the workflow lifecycle.
- **Two event-bus backends.** Redis/BullMQ for production (durable queue, retries) and local `EventEmitter` for dev — both behind the same `IEventBusModuleService` (`research/medusa/packages/modules/event-bus-redis/src/services/event-bus-redis.ts:50`, `research/medusa/packages/modules/event-bus-local/src/services/event-bus-local.ts:24`). Neither offers exactly-once; at-least-once with handler-level idempotency is the model.
- **Cross-module saga example.** `completeCart` payment-refund compensation resolves `Modules.PAYMENT` from the container and invokes `refundPaymentAndRecreatePaymentSessionWorkflow` if the forward step failed (`research/medusa/packages/core/core-flows/src/cart/steps/compensate-payment-if-needed.ts:26`). This is the canonical cross-module pattern — and is **precisely the shape rntme will delegate to Zeebe**, not implement.

## Gaps for commerce-class case

### [P0] [demo-blocker] Intra-service multi-aggregate command with transactional guarantee

**Why critical / DX impact.**
The `checkoutCart` command in the commerce demo spec touches **two aggregates in one service**: it must mutate `Cart` (transition to `completed`) and create `Order` atomically. Today rntme cannot express this: `compileCommand` rejects any graph whose `emit` nodes span >1 aggregate, so the LLM agent is forced to either (a) collapse Order into Cart (wrong domain model), (b) emit from outside the command graph via a hand-rolled script (unauthoritative, unobservable), or (c) pretend the cross-aggregate transition is a Zeebe saga, which overcorrects — this is intra-service, same SQLite, same transaction scope. Without this, the commerce demo's central workflow cannot be authored as an artifact at all.

**Pain point in rntme today** (concrete pattern/line).
`packages/graph-ir-compiler/src/command-runtime/compile.ts:80` explicitly returns `CMD_MULTI_AGGREGATE_NOT_ALLOWED` when `new Set(emitPlans.map(e => e.aggregate)).size !== 1`. Downstream, `packages/graph-ir-compiler/src/command-runtime/execute.ts:61` bakes in the single-stream assumption: `head = compiled.emits[0]!`, one `stream`, one `readStream`+replay, one `AppendRequest`. The append API itself *does* accept `readonly AppendRequest[]` and wraps them in one SQLite transaction (`packages/event-store/src/store/sqlite.ts:51`), so the **event-store layer can already do multi-stream atomic append** — the bottleneck is purely in the command compiler/executor, which only constructs one request.

**Medusa reference**.
Medusa handles this with a workflow of two steps plus an `eventGroupId` — step 1 creates the order, step 2 transitions the cart, both staged events released together (`research/medusa/packages/core/types/src/event-bus/event-bus-module.ts:66`, `research/medusa/packages/core/orchestration/src/transaction/distributed-transaction.ts:121`). Note that Medusa *also* uses workflow-level compensation here, because Cart and Order might live in different modules. In rntme they live in the same SQLite database for one service, so the need for compensation collapses: we can just atomic-append two streams.

**Authorability / visualization** — how closing this gap affects what the LLM can author and what the business-user UI shows.
With multi-aggregate commands, the LLM can emit a `checkoutCart.json` graph with two `emit` nodes (`emit.cartCompleted`, `emit.orderPlaced`) wired to the same DAG, and the compiler builds one composite `AppendRequest[]` per aggregate — guaranteed-atomic at the SQLite level without a workflow runtime. The visualization UI renders the command-graph as a fan-out: a single "Checkout" input feeds two terminal emit nodes with a shared "transactional boundary" overlay — exactly the mental model a reviewer needs. Without this, the UI can only ever show one emit per command, and the checkout flow has to be drawn as two disconnected commands with a hand-annotated "both must succeed" caption.

#### Intra-service multi-aggregate command shape

This sub-section tracks the three shape options from spec §8 question 1. Decision deliberately left open; each candidate will be weighed during graph-IR v2 design.

**Option a. Extended `emit` → DAG-emit with transactional guarantee.**
- *Pros:* minimum new surface area — `emit` already exists, lift the single-aggregate check at `compile.ts:80`, let the executor construct `AppendRequest[]` with one entry per aggregate, and rely on `SqliteEventStore.appendEvents` which already wraps in one transaction. Stays artifact-authorable (one graph, two emit nodes). Visualization stays inside the existing command-graph canvas.
- *Cons:* per-aggregate OCC becomes composite (either all `expectedVersion`s match or the whole transaction retries); error-surface grows — a `ConcurrencyConflict` on aggregate B after aggregate A has been replayed forces the executor to re-read both streams. Partial-result semantics (aggregate A replayed successfully but B failed mid-flight) need clear error codes.

**Option b. Separate "saga-step" primitive for intra-service only.**
- *Pros:* mirrors Medusa's step/compensation model and gives a natural path to per-step retries + per-step compensation; explicit about invoke/compensate pairs. Keeps `emit` single-aggregate (no regression risk). Useful if we ever allow a step to call an external stub (payment) before the Zeebe migration.
- *Cons:* introduces a second command artifact kind, doubling the artifact surface the LLM must learn and the UI must render. Collides with Zeebe's territory — if the only place saga-step shows up is intra-service, we pay the abstraction cost without getting the cross-service payoff. Risks encouraging authors to reach for it when they should be reaching for Zeebe.

**Option c. Outbox + manual compensation.**
- *Pros:* each aggregate mutated by a **separate** command; emit events and let a consumer (or a downstream command triggered by a projection watcher) close the loop. No change to graph-IR compiler; purely a pattern doc + a reliable outbox.
- *Cons:* abandons intra-service atomicity entirely — Cart transitions to completed but Order creation is eventually-consistent, so a crash between the two leaves an invariant-violating state visible to readers. Compensation is manual per-flow and therefore LLM-unfriendly (each commerce author must hand-roll it). Fails the "visually verify commerce logic" criterion because the reviewer sees two disconnected commands and a comment, not one transactional boundary.

Decision path: option (a) is the cheapest and aligns with the fact that the event-store already supports multi-stream atomic append. Options (b) and (c) will be revisited during graph-IR v2 design — in particular if we find that intra-service multi-aggregate commands actually want step-level retry semantics (a signal to reconsider (b)) or if we find that the OCC composite-retry story is ugly (a signal to reconsider (c)).

### [P0] [non-blocker] Outbox pattern for reliable event emission

**Why critical / DX impact.**
A commerce service that emits `OrderPlaced` must guarantee the event reaches Kafka *exactly once per committed append* — downstream services (inventory, payments, ksqlDB analytics) cannot tolerate silent drops, and duplicate emissions inflate metrics and trigger double-charge defenses. Today the relay is at-least-once by construction (send-then-advance-cursor), and a crash between `kafka.send` success and `writeCursor` re-emits on restart. For the demo this is fine (consumers are idempotent), but the absence of an explicit **outbox artifact** means an LLM agent authoring a new service has no declarative knob to say "this command's emitted events must be durably queued for Kafka" — it's a runtime behaviour hidden in the relay loop.

**Pain point in rntme today** (concrete pattern/line).
`packages/event-store/src/relay/loop.ts:47` is the canonical at-least-once loop: iterate records, send to Kafka, advance `highestDeliveredId` per success, write cursor after the batch. The cursor is updated *after* all sends complete, so a crash mid-batch re-sends the whole batch. There is no dedupe table on the producer side and no `outbox` vs. `event_log` separation — the event log *is* the outbox. That works, but it prevents future features like "emit X to Kafka topic A, but don't emit Y" (selective outbox), and it has no explicit `published_at` / `send_attempts` telemetry.

**Medusa reference**.
Medusa's Redis event bus uses BullMQ job persistence with automatic retries — the job queue **is** the outbox (`research/medusa/packages/modules/event-bus-redis/src/services/event-bus-redis.ts:50`). Grouped events (`eventGroupId`) are Medusa's cross-module transactional-outbox: events are staged in `groupedEventsMap_` and only handed to BullMQ after `releaseGroupedEvents(groupId)` (`research/medusa/packages/modules/event-bus-local/src/services/event-bus-local.ts:24`, `research/medusa/packages/core/types/src/event-bus/event-bus-module.ts:66`). The design principle — **stage inside the transaction, publish after commit** — maps cleanly onto rntme's SQLite-then-Kafka path.

**Authorability / visualization** — how closing this gap affects what the LLM can author and what the business-user UI shows.
Making the outbox explicit (either a dedicated `outbox` table distinct from `event_log`, or an artifact-level `emit.publish: boolean` flag per event-type) lets the LLM express "this domain event is internal / not wired to Kafka" vs. "this event is a public contract". The business-user UI can then mark events with a "published to Kafka" badge on the aggregate timeline, and the relay's health metrics (queue depth, retry count, send failures) become visualisable per event-type — not just a single "relay is behind" gauge.

### [P1] [non-blocker] Exactly-once semantics in projection-consumer

**Why critical / DX impact.**
Today projection-consumer is at-least-once + idempotent — safe, but requires every `applyEvent` path to carry a version check and every INSERT to use `ON CONFLICT` or a version guard. That is fine for the entity-mirror pattern (one aggregate → one row, version monotonic per row), but as QSM grows multi-row derivations (counters, running totals, windowed aggregates) the "just check version" trick breaks: a counter increment is not idempotent under replay. A commerce-class QSM that needs "open-orders count" or "hourly GMV" needs either exactly-once delivery or explicit stateful dedupe per handler.

**Pain point in rntme today** (concrete pattern/line).
`packages/projection-consumer/src/apply/apply-event.ts:26` hinges idempotency on a per-row `last_event_version` column — it works for entity-mirror but does not generalise. `packages/projection-consumer/src/consumer.ts:30` does `BEGIN IMMEDIATE → apply → COMMIT → commitOffsets`; the commit-offsets happens outside the DB transaction, so a crash between COMMIT and offset-commit re-delivers the batch. There is no per-batch processed-offsets table on the consumer side to absorb redeliveries generically.

**Medusa reference**.
No direct Medusa analogue — Medusa's event handlers are explicitly at-least-once with handler-level idempotency. The best parallel is `TransactionContext.invoke` which memoises step results keyed by step id so replays return cached outputs (`research/medusa/packages/core/orchestration/src/transaction/distributed-transaction.ts:85`). For QSM we'd want the consumer-side equivalent: a `(topic, partition, offset)` → `applied` table checked before each apply.

**Authorability / visualization** — how closing this gap affects what the LLM can author and what the business-user UI shows.
Closing this unlocks **non-idempotent projections** in QSM — counters, sliding-window aggregates, top-N leaderboards — which commerce dashboards routinely need (e.g., "orders placed in last hour" as a projection, not a live aggregation query). The UI can then render a "projection lag + exactly-once healthy" indicator per QSM entity-mirror, and the LLM can author QSM handlers without having to prove idempotency for every derived column. Without this, QSM stays entity-mirror-only and richer read models have to move to ksqlDB (see Intersections).

### [P1] [non-blocker] Idempotency on API-level command retries

**Why critical / DX impact.**
A POST `/v1/carts/:id/checkout` is almost always retried by the client on network blip. Today rntme has zero request-level dedupe: the client retries, the server executes the command twice, the event-store's OCC catches the second attempt (good — `COMMAND_CONCURRENCY_CONFLICT` returned), but the client sees an error instead of the original success. For non-mutating retries this is fine; for checkouts it means the client must parse the error and query state to know whether the original succeeded — a pattern every LLM agent will get wrong. Cross-link: this gap intersects directly with bindings HTTP behaviour (see `bindings-gaps.md` for the HTTP-header / OpenAPI side).

**Pain point in rntme today** (concrete pattern/line).
`demo/issue-tracker-api/src/server.ts:45` wires `createBindingsRouter` directly to `executeCommand` with no middleware reading an `Idempotency-Key` header, no request cache, no response replay. The command runtime `executeCommand` itself has no key parameter (`packages/graph-ir-compiler/src/command-runtime/execute.ts:42`); re-running the same command with the same inputs produces a second event unless OCC happens to catch it (which it only does if the aggregate state changed as a consequence of the first call — fine for state-machine transitions, broken for idempotent-by-domain operations like "add line item").

**Medusa reference**.
Medusa propagates `idempotency_key` through `TransactionMetadata` — available to every step via `context.idempotencyKey` (`research/medusa/packages/core/workflows-sdk/src/utils/composer/helpers/create-step-handler.ts:12`). Workflow checkpoint storage keys on `(idempotencyKey, model_id)`, so a re-run with the same key resumes the previous execution and returns the cached result instead of re-doing side effects (`research/medusa/packages/core/orchestration/src/transaction/base-in-memory-storage.ts:7`). Some workflows mark `idempotent: false` to signal side effects (payment capture) where retries must be blocked, not resumed.

**Authorability / visualization** — how closing this gap affects what the LLM can author and what the business-user UI shows.
With an idempotency-key cache keyed on `(commandId, Idempotency-Key)` → previous `CommandResult`, the LLM can author binding specs that reliably survive client retries and the UI can surface "retried once, same outcome" on the aggregate timeline instead of "error: concurrency conflict". For commerce demos this is table-stakes: the checkout screen's retry button has to be safe. The binding-level side (OpenAPI `parameters`, response caching) is tracked in `bindings-gaps.md`; this entry focuses on the command-runtime contract — `executeCommand` needs to accept and respect an `idempotencyKey` parameter.

### [P2] [non-blocker] Scheduled / cron jobs primitive

**Why critical / DX impact.**
Commerce flows need time-based commands: abandon-cart sweep after 24h, subscription-renewal billing, token-refresh before expiry. Today rntme has no artifact that says "fire command X every N minutes" or "fire at cron schedule Y" — the only trigger is inbound HTTP. An LLM agent modelling a subscription-billing service cannot express the billing cadence in an artifact at all. This is **tentative**: if Zeebe owns scheduled triggers at the platform level (timer catch events on BPMN), rntme may never want this primitive. Current leaning: **defer pending the Zeebe scheduling decision**, and if Zeebe timer events can trigger rntme commands via gRPC, this gap is closed-as-out-of-scope.

**Pain point in rntme today** (concrete pattern/line).
`demo/issue-tracker-api/src/server.ts:42` starts only a HTTP-reactive pipeline; there is no scheduler, no `setInterval`-wrapped command runner, no artifact that declares a cron expression. The event pipeline is inbound-only: `buildEventPipeline().start()` spins up the relay and consumer but no producer-of-commands.

**Medusa reference**.
Medusa's scheduled jobs live in a dedicated `ScheduledJob` module (cron-driven, separate from the workflow engine) — not directly surveyed in survey B; the equivalent composable primitive is `WorkflowManager` paired with an external cron, not an in-workflow timer. The cleaner analogue is BPMN timer events (Zeebe's territory), which is why this is *likely out-of-scope* for rntme.

**Authorability / visualization** — how closing this gap affects what the LLM can author and what the business-user UI shows.
If kept in rntme: the LLM declares `scheduledCommands: [{ commandId: 'sweepAbandonedCarts', cron: '0 * * * *' }]` in a service artifact, and the UI renders a "scheduled" lane next to the command list, with last-run / next-run tiles. If punted to Zeebe: rntme exposes a `/grpc/internal/executeCommand` endpoint consumable by Zeebe timer events, and the authoring layer is Zeebe BPMN, not a rntme artifact — which is arguably better because the scheduling graph and the business-process graph become one artifact.

## Intersections with out-of-scope

- **Cross-service saga orchestration → Zeebe** (not in rntme). Medusa's `completeCart` payment-refund compensation (`research/medusa/packages/core/core-flows/src/cart/steps/compensate-payment-if-needed.ts:26`) is the canonical cross-module saga and is **exactly** the shape that moves to Zeebe in the rntme platform vision. rntme services expose gRPC command endpoints; Zeebe BPMN processes drive invoke + compensate across services. The intra-service multi-aggregate remnant (P0 above) is what stays in rntme — everything wider is Zeebe's job.
- **Scheduled jobs at platform level → Zeebe timer events** (may deprecate P2 scheduled-jobs gap above). BPMN timer catch events plus gRPC-to-rntme commands give us a cleaner authoring surface than duplicating cron inside rntme. Final decision is tracked in Open Questions below.
- **Event transformation / enrichment between services → ksqlDB**. Medusa's grouped events released to handlers that enrich data from other modules (`research/medusa/packages/core/types/src/event-bus/event-bus-module.ts:66`) do not map to rntme — our projection-consumer is strictly single-service entity-mirror. Cross-service joins, analytics, windowed aggregates, and event enrichment (e.g., joining `OrderPlaced` with `CustomerCreated` from another service) live in ksqlDB streams, not in QSM.
- **Exactly-once in Kafka itself.** The P1 consumer-side exactly-once gap above is scoped to the projection-consumer. True end-to-end exactly-once (producer idempotence + transactional writes to Kafka) is a Kafka-config concern and sits outside rntme's artifact model — the relay can be upgraded to use Kafka transactional producers without changing any artifact.
- **Workflow-level compensation**. Medusa's `StepResponse.permanentFailure(..., compensateInput)` and the reverse-topological-order compensator live at a higher level than rntme aims to cover. For intra-service multi-aggregate (P0), the SQLite transaction *is* the compensation — either both aggregates appended or neither. For cross-service flows, Zeebe owns compensation boundary events. There is no rntme-level compensation primitive, deliberately.

## Open questions

- **If Zeebe handles scheduled jobs, do we need a scheduled-jobs primitive in rntme at all, or only Zeebe-triggered commands?** Two sub-questions: (1) Does every rntme service expose a gRPC command endpoint that Zeebe can invoke on a timer event? (If yes, the P2 gap closes as out-of-scope.) (2) Are there service-local scheduled tasks (e.g., a per-service materialized-view refresh) that shouldn't cross the Zeebe boundary? Current leaning: punt to Zeebe; only add scheduled-commands to rntme if we find a class of tasks that genuinely want per-service scheduling without a BPMN process.
- **Should the multi-aggregate P0 gap pick shape option (a), (b), or (c)?** See sub-section above. Decision blocked until graph-IR v2 design brief lands — needs input from the OCC-on-composite story (can we cleanly retry when aggregate B has moved?), and from the LLM-authorability story (do authors struggle with two emit nodes in one graph or find it natural?).
- **Should `executeCommand` accept `idempotencyKey` as a first-class parameter (P1 above), or should dedupe live purely in the HTTP binding layer?** Command-runtime parameter is cleaner conceptually (idempotency is a command contract, not a transport concern) but forces every caller — tests, CLI, future gRPC — to plumb it. Binding-layer is narrower (only HTTP cares) but doesn't protect non-HTTP callers. Cross-decision with `bindings-gaps.md`.
- **Does the outbox deserve its own table, or is `event_log` + `publish_cursor` already the outbox?** Functionally equivalent; the separation matters only if we want to express "emit this event but don't publish it to Kafka" (internal-only events). Open until a concrete non-publishable event-type appears in a real service.
