# @rntme/storage-s3

S3-compatible vendor module for the canonical Storage contract `@rntme/contracts-storage-v1`. It covers AWS S3, Cloudflare R2, MinIO, rustfs, DigitalOcean Spaces, Backblaze B2, Tigris, and compatible backends through one module manifest and env mapping.

## File map

```
modules/storage/s3/
├── module.json
├── Dockerfile
├── src/
│   ├── bin/server.ts             Bun server entry; reads env, storage.json, and SQLite path.
│   ├── server.ts                 gRPC StorageModule service definition and server factory.
│   ├── handler.ts                Seven Storage RPC handlers.
│   ├── s3-client.ts              Bun.s3 env mapping and S3ClientLike adapter.
│   ├── pending-store.ts          SQLite pending/idempotency store.
│   ├── route-resolver.ts         storage.json route lookup and content rules.
│   ├── sweeper.ts                Pending upload expiry sweep.
│   ├── provisioner/              Node-only AWS SDK provisioner.
│   └── client/                   Browser boot, operations, and React file components.
└── test/
    ├── unit/                     Handler, provisioner, client, and support tests.
    └── integration/              Testcontainers RustFS/MinIO checks; set SKIP_INTEGRATION=1 to skip.
```

## Quick start

Build all three runtimes:

```bash
bun run build
```

Minimal project module reference:

```json
{
  "modules": {
    "storage": {
      "module": "@rntme/storage-s3",
      "publicConfig": {
        "bucketName": "my-app-files",
        "region": "us-east-1",
        "backend": "aws-s3",
        "appOrigins": ["https://app.example"]
      }
    }
  }
}
```

Per-service upload policy lives in `services/<svc>/storage.json`; `@rntme/blueprint` validates route ids, PDM owner bindings, MIME/size/count rules, and lifecycle durations before composition.

## Deploy workload image

`@rntme/storage-s3` is an `integration-module` service kind. The runnable workload image is configured on the deploy target at `target.modules.storage-s3.image`; `@rntme/deploy-core` planning fails with `DEPLOY_PLAN_MISSING_MODULE_IMAGE` when that entry is absent. Render emits one `mod-storage-s3` workload per project-stack Compose resource, and the runtime dials it at `mod-storage-s3:50051` for graph IR `call` nodes whose `target.module` resolves to the project's `storage` module key.

## API

RPCs implemented by `src/handler.ts`:

- `PrepareUpload`
- `CommitUpload`
- `AbortUpload`
- `GetFile`
- `ListFiles`
- `GetDownloadUrl`
- `DeleteFile`

Events published through the injected bus:

- `FileUploadInitiated`
- `FileUploadCommitted`
- `FileUploadAborted`
- `FileOrphaned`
- `FileDeleted`
- `FileLifecycleSwept`

Browser client exports:

- `boot(ctx)` registers five data-returning operations through the client-runtime registry.
- `<UploadDropzone/>` uses Uppy + AwsS3 for direct PUT uploads.
- `<FileList/>` lists and deletes files for a route/entity pair.
- `<FilePreview/>` refreshes presigned GET URLs on `document.visibilityState === "visible"`.

Provisioner exports:

- Auto mode when `targetSecrets.s3Admin` exists: reconcile bucket, CORS, lifecycle, and AWS IAM scoped credentials where the backend supports IAM.
- Manual mode when `targetSecrets.s3Scoped` exists: validate an existing bucket using scoped credentials.

## Invariants & gotchas

- Server runtime is Bun-only. `src/bin/server.ts` uses `bun:sqlite` and the Bun S3 client surface.
- Provisioner runtime is Node-only and bundled to `dist/provisioner.entry.js` with Bun.
- Browser runtime imports only `@rntme/contracts-client-runtime-v1` plus React/Uppy dependencies; it does not import server or provisioner code.
- `PrepareUpload` is idempotent for 24 hours on `(routeId, entityId, idempotencyKey)` and returns a fresh presigned URL on replay.
- Default presign TTL is 900 seconds.
- Pending upload TTL is route-configurable through `storage.json#routes.<id>.lifecycle.expirePending`.
- Provisioned RustFS uses separate internal and public endpoints: server-side S3 calls use `STORAGE_S3_ENDPOINT`, browser presigned URLs use `STORAGE_S3_PUBLIC_ENDPOINT`.
- When `STORAGE_S3_BACKEND=rustfs`, server startup idempotently ensures the project bucket, CORS, and supported lifecycle rules. This is an MVP bridge until deploy can apply infrastructure before running module provisioners.
- Backend IAM varies. AWS S3 supports scoped IAM creation; R2, MinIO, rustfs, Spaces, B2, and Tigris use fallback/manual scoped credentials unless a backend-specific IAM API is added.
- Use distinct container DNS names and data volumes when running multiple storage stacks in the same Dokploy project.

## Out of scope

- Multipart uploads.
- Content-addressed deduplication.
- Server-side file processing.
- Webhook-based commit.
- Orphan scanning against deleted aggregate rows; v1 only defines the event surface.

## Where to look first

- `src/handler.ts` — RPC orchestration and event publishing.
- `src/provisioner/index.ts` — auto/manual entrypoint and AWS SDK adapters.
- `src/provisioner/admin-mode.ts` and `src/provisioner/manual-mode.ts` — provisioner mode details.
- `src/client/upload-dropzone.tsx`, `src/client/file-list.tsx`, `src/client/file-preview.tsx` — browser UI.
- `test/unit/handler.test.ts` — expected core RPC behavior.
- `test/integration/` — RustFS/MinIO checks when a Testcontainers runtime is available.
