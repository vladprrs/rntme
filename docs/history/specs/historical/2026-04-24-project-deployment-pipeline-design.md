# Project Deployment Pipeline — design

**Status:** design approved in brainstorming, awaiting user review of this spec
**Author:** brainstorm 2026-04-24

- `docs/history/specs/historical/2026-04-15-runtime-packaging-design.md` — current one-service runtime packaging.
- `docs/history/specs/historical/2026-04-19-platform-api-design.md` — platform registry and validation gate; deploy-controller is out of scope there.

## 1. Problem

The project-first blueprint model makes a project the authoring unit, but the
current runtime still starts one service at a time. A useful project deployment
needs to materialize several domain services, integration modules, shared edge
routing, and target infrastructure as one coherent deployment.

The deployment layer must also fit two different callers:

- a local CLI flow where an agent has a project blueprint folder and target
  credentials;
- a future platform flow where the platform has already accepted a validated
  project version, stores target credentials, and exposes a "Deploy" action in
  the UI.

The deployment layer should not become a new project runtime that owns business
execution. Business execution remains inside service runtimes and integration
module containers.

## 2. Goal

Define a **Project Deployment Pipeline** made of:

- `rntme-cli/packages/deploy-core` — target-neutral deployment planning from a
  validated/composed project model;
- `rntme-cli/packages/deploy-dokploy` — first target adapter, rendering and
  applying the plan to Dokploy through the Dokploy HTTP API;
- CLI/platform frontends — collect config/secrets, call plan/render/apply,
  persist redacted audit artifacts, and run post-deploy verification.

The MVP target is a Dokploy preview deployment:

- domain services run as separate `@rntme/runtime` containers;
- integration modules run as separate containers when image config is provided;
- a generated Nginx container implements project edge routing and supported
  middleware;
- persistence is ephemeral;
- production mode exists in the model but is rejected until runtime production
  prerequisites are implemented.

## 4. Scope

### 4.1 In scope

- Target-neutral deployment plan model.
- Deployment-specific validation over a composed project.
- Dokploy rendered plan model.
- Dokploy apply contract through the Dokploy HTTP API.
- Nginx edge route and middleware rendering.
- Domain service artifact mini-image build context generation inside
  `deploy-dokploy`.
- Integration module image config and exposure rules.
- Redacted render/apply artifacts for audit/debug.
- Frontend verification contract and expected `DeploymentApplyResult` shape.

### 4.2 Out of scope

- A new project runtime process.
- Runtime business execution changes.
- Deploy-controller/reconciler/status loop.
- Platform credentials storage UI.
- Automatic rollback.
- Managed Redpanda provisioning.
- Operaton/BPMN provisioning.
- Production persistent storage.
- Auth middleware execution.
- Replacing `InMemoryBus` in `@rntme/runtime`; this is a required follow-up.
- Writing committed UI tests for deployed apps.

## 5. Package Boundaries

### 5.1 `deploy-core`

`deploy-core` owns target-neutral planning.

It accepts:

- a validated/composed project model from `@rntme/blueprint`;
- a typed deployment config object;
- target-independent module/workload declarations.

It returns:

- `ProjectDeploymentPlan`, or deployment validation errors.

It does not:

- read raw blueprint folders;
- call `loadComposedBlueprint`;
- read environment variables;
- read secrets from disk or platform storage;
- generate Dockerfiles;
- know about Dokploy resource types;
- run browser verification.

The local CLI can expose a convenience flow of `folder → validate/compose →
plan`, but the validation/compose step lives in the caller/frontend layer.

### 5.2 `deploy-dokploy`

`deploy-dokploy` owns Dokploy-specific rendering and apply.

It accepts:

- `ProjectDeploymentPlan`;
- typed Dokploy target config;
- in-memory secret values for apply only.

It returns:

- redacted rendered Dokploy plan;
- structured apply result or partial failure report.

It does not:

- run blueprint validation;
- store platform credentials;
- call browser tooling;
- use Dokploy MCP as a package API.

MCP can still be used by operators or agents for manual debugging, but the
library contract is the Dokploy HTTP API.

### 5.3 CLI and platform frontends

Frontends own orchestration around the libraries:

- collect target config and credentials;
- call validation/composition when deploying a local folder;
- call `deploy-core.plan`;
- call `deploy-dokploy.render`;
- show preview/diff;
- call `deploy-dokploy.apply`;
- persist redacted render/apply artifacts;
- run verification, including `agent-browser` UI smoke checks.

The future platform UI flow uses the same packages. The platform gathers target
credentials/config from its own storage and passes typed objects to the
pipeline.

## 6. Deployment Modes

### 6.1 `preview`

`preview` is the only supported MVP mode.

Properties:

- domain service persistence is ephemeral;
- service/runtime containers are single-instance;
- edge gateway is Nginx;
- supported middleware is limited to the MVP list;
- post-deploy smoke verification is expected;
- deploy is suitable for demos, previews, and agent validation loops.

### 6.2 `production`

`production` is present in types and user-facing config, but MVP planning
rejects it.

Required follow-ups before enabling production:

- replace the temporary runtime `InMemoryBus` default with a production Kafka or
  Redpanda-capable runtime plugin/package;
- define persistent storage for service event-store and QSM databases;
- decide how target deployments receive or mount production artifacts;
- define auth middleware execution at the gateway or via an external auth
  service;
- define deployment records, status tracking, and rollback semantics.

This spec intentionally records those requirements instead of silently treating
preview deploys as production-ready.

## 7. Target-Neutral Plan

### 7.1 Shape

Illustrative TypeScript shape:

```ts
export type ProjectDeploymentPlan = {
  project: {
    orgSlug: string;
    projectSlug: string;
    environment: 'default';
    mode: 'preview' | 'production';
  };
  infrastructure: {
    eventBus: ExternalEventBusRequirement;
  };
  workloads: DeploymentWorkload[];
  edge: EdgePlan;
  diagnostics: {
    warnings: DeploymentWarning[];
  };
};
```

`environment` is part of the model now, but MVP only accepts `"default"`.
Resource names do not include environment in MVP.

### 7.2 Infrastructure

The MVP infrastructure requirement is one project-level external event bus:

```ts
export type ExternalEventBusRequirement = {
  kind: 'kafka';
  mode: 'external';
  brokers: string[];
  topicPrefix?: string;
  security?: {
    protocol: 'plaintext' | 'sasl_ssl';
    secretRefs?: Record<string, string>;
  };
};
```

The plan has no per-service event bus override in MVP.

The current runtime defaults to `InMemoryBus`. This is a known gap. The
deployment pipeline model is production-shaped around external Redpanda/Kafka,
but production planning remains rejected until a runtime Kafka bus plugin/image
exists.

### 7.3 Workloads

`domain-service`:

- corresponds to a `domain` service in the composed project;
- runs a service-level `@rntme/runtime` container;
- receives service artifacts generated from the composed project;
- receives project-level external event bus env/config;
- uses ephemeral persistence in `preview`.

`integration-module`:

- corresponds to an `integration` service such as `mod-workos`;
- requires explicit image config in deployment config;
- may be internal-only;
- may receive a public route only when its module config sets `expose: true`.

`edge-gateway`:

- corresponds to generated Nginx;
- receives public traffic from Dokploy/Traefik;
- routes to domain services and explicitly exposed integration modules;
- executes supported middleware.

Every service in the composed project must map to a deployable workload. An
integration module without image config is a deployment validation error.

### 7.4 Naming and labels

Target resource names use:

```text
rntme-<orgSlug>-<projectSlug>-<workloadSlug>
```

`orgSlug` comes from deployment config in the local MVP. The project slug comes
from the project model. `environment: "default"` is stored in target
labels/metadata, not in the resource name.

Adapters may add a short hash suffix only when required by target name length or
character limits. If they do, the rendered plan must preserve the original
logical name and the final target name.

## 8. Edge Routing and Middleware

### 8.1 Routes

The plan carries public UI and HTTP routes from the composed project routing
context.

Domain services may receive public UI/API routes according to `project.json`.

Integration services may receive public routes only when the module deployment
config includes:

```ts
{
  expose: true
}
```

A public route to an integration module without explicit exposure is a
deployment validation error.

### 8.2 Supported middleware

Preview MVP supports:

- `request-context` — Nginx adds or forwards request/correlation headers;
- `rate-limit` — Nginx renders rate limit zones and route-level limit rules;
- `body-limit` — Nginx renders `client_max_body_size`;
- `timeout` — Nginx renders upstream/read/send timeout settings.

Unsupported middleware is a deployment validation error. This includes `auth`.

The deployment layer must not silently deploy a route that declares auth while
leaving it open.

### 8.3 Policy values

`project.json` declares middleware intent and mount points. Deployment config
provides environment-specific values.

Example:

```json
{
  "policies": {
    "rateLimit": {
      "default": { "requestsPerMinute": 60, "burst": 20 }
    },
    "bodyLimit": {
      "default": { "maxBodySize": "2m" }
    },
    "timeout": {
      "default": { "upstreamTimeoutMs": 30000 }
    }
  }
}
```

If a project middleware declaration references `policy: "default"` and the
deployment config does not define it, planning fails with a deployment
validation error.

## 9. Dokploy Rendering

### 9.1 Target config

Dokploy config supports:

- `dokployProjectId` — preferred for platform-managed flows;
- or `dokployProjectName` with `allowCreateProject: true` — allowed for local
  CLI workflows.

Without a project id and without allowed project creation, render/apply fails
validation.

### 9.2 Render output

`deploy-dokploy.render(plan, dokployConfig)` returns a redacted rendered plan.

The rendered plan includes:

- target project lookup/create instruction;
- Dokploy applications for domain services;
- Dokploy applications for integration modules;
- Dokploy application for Nginx edge;
- generated Nginx config description;
- generated Docker build context descriptors;
- env var names and secret refs;
- target labels for idempotent lookup;
- rendered plan digest;
- warnings.

The rendered plan does not include secret values.

### 9.3 Domain artifact images

Domain services use a common `rntme-runtime` base image, but the Dokploy adapter
generates a service-specific artifact mini-image/build context.

This keeps the service runtime standardized while giving Dokploy an immutable,
deployable artifact per service.

`deploy-core` remains Docker-agnostic. It describes which service artifacts a
workload needs; `deploy-dokploy` decides how to package those artifacts.

### 9.4 Integration module images

Integration modules use explicit image config.

The MVP does not auto-build vendor modules or infer module images from
`packages/module-skeleton`. If a project contains an integration service, its
deployment config must say how to run it.

### 9.5 Edge image

The edge workload uses an Nginx image plus generated config. Dokploy/Traefik
terminates public TLS and routes traffic to this internal edge application.

Nginx then applies project routes and supported middleware before proxying to
service/module upstreams.

## 10. Apply Semantics

### 10.1 Apply input

`deploy-dokploy.apply(renderedPlan, secrets)` receives secret values only in
memory. Rendered plans, results, logs, and audit artifacts must use secret refs
or redacted names only.

### 10.2 Idempotency

Apply uses deterministic names and labels to upsert resources.

Repeated apply of the same rendered plan should converge the Dokploy project to
the requested state. The future platform can store returned resource ids in a
deployment record, but the MVP does not require a state store.

### 10.3 Rollback

MVP has no automatic rollback.

If apply fails partway through, it returns a structured partial failure report
that includes:

- which resources were created;
- which resources were updated;
- which step failed;
- a machine-readable error code/message;
- whether retry is expected to be safe.

The recovery path is to fix config/target state and rerun idempotent apply.

## 11. Apply Result

`DeploymentApplyResult` is structured enough for CLI verification, platform UI,
and audit logs.

Illustrative shape:

```ts
export type DeploymentApplyResult = {
  target: {
    kind: 'dokploy';
    projectId: string;
  };
  deployment: {
    orgSlug: string;
    projectSlug: string;
    environment: 'default';
    mode: 'preview';
  };
  resources: Array<{
    logicalId: string;
    workloadSlug: string;
    kind: 'domain-service' | 'integration-module' | 'edge-gateway';
    targetResourceId: string;
    targetResourceName: string;
    action: 'created' | 'updated' | 'unchanged';
  }>;
  urls: {
    projectUrl?: string;
    uiUrl?: string;
    publicRoutes: Array<{ routeId: string; url: string }>;
    internalServices: Array<{ serviceSlug: string; url: string }>;
  };
  renderedPlanDigest: string;
  warnings: DeploymentWarning[];
  verificationHints: DeploymentVerificationHints;
};
```

No secret values appear in the result.

## 12. Audit and Debug Artifacts

After render/apply, the caller should persist:

- redacted rendered target plan;
- rendered plan digest;
- apply result;
- warnings and partial failure details when present.

The MVP does not persist full generated Docker contexts or unredacted Nginx
config archives as audit artifacts. If diagnostic output is needed, it must be
redacted before being written to logs or platform storage.

## 13. Verification

Verification lives in the frontend layer, not in `deploy-core` or
`deploy-dokploy`.

### 13.1 Backend smoke

After apply, the CLI/platform should verify:

- edge health endpoint responds;
- public UI URL responds when present;
- public service route smoke succeeds, using routed health/openapi endpoints
  where available;
- failed checks produce a structured verification report.

### 13.2 UI smoke

The frontend should use `agent-browser` for deployed UI smoke:

- open the public UI URL;
- confirm the SPA loads and the first screen is visible;
- perform one or two basic clicks on interactive controls;
- observe navigation/API errors if exposed by the browser automation workflow.

This is a deploy verification workflow/checklist. It is not a committed UI test
suite in this spec.

## 14. Error Handling

Deployment APIs return `Result<T>` in the workspace style:

```ts
{ ok: true, value: T } | { ok: false, errors: DeploymentError[] }
```

Error codes should be stable and append-only. Suggested prefixes:

- `DEPLOY_PLAN_*` for target-neutral planning errors;
- `DEPLOY_RENDER_DOKPLOY_*` for Dokploy rendering errors;
- `DEPLOY_APPLY_DOKPLOY_*` for Dokploy API/apply errors;
- `DEPLOY_VERIFY_*` for frontend verification reports, if modeled in a package.

Representative errors:

- unsupported production mode;
- missing `orgSlug`;
- non-default environment in MVP;
- missing external event bus config;
- unsupported middleware;
- missing middleware policy value;
- integration module missing image config;
- public integration route without `expose: true`;
- missing Dokploy project id or create permission;
- secret value found in rendered output;
- partial apply failure.

## 15. Testing

Minimum test coverage for implementation:

- `deploy-core` unit tests for preview plan construction from composed project
  fixtures.
- `deploy-core` validation tests for rejected production mode.
- `deploy-core` validation tests for missing event bus config.
- `deploy-core` validation tests for integration module missing image config.
- `deploy-core` validation tests for public integration route without
  `expose: true`.
- `deploy-core` validation tests for unsupported middleware such as `auth`.
- `deploy-core` validation tests for missing policy values.
- `deploy-dokploy` render tests for deterministic resource names and labels.
- `deploy-dokploy` render tests for redacted secrets.
- `deploy-dokploy` render golden test for Nginx config shape.
- `deploy-dokploy` apply tests with a fake Dokploy client covering create,
  update, unchanged, and partial failure.
- CLI/platform verification can be covered by smoke workflow docs initially;
  browser automation is run manually/post-apply rather than committed as UI
  tests in this spec.

## 16. Follow-ups

Follow-up specs or plans should cover:

- Kafka/Redpanda runtime bus plugin and runtime manifest/env changes.
- Persistent service storage for Dokploy preview-to-production progression.
- Managed Redpanda provisioning for targets that support it.
- Operaton/BPMN provisioning and wiring.
- Auth middleware execution through Nginx `auth_request`, Kong, or another
  explicit edge contract.
- Platform deployment records, credentials storage, status loop, and button
  deploy UX.
- Reconciler/drift detection.
- Rollback to previous rendered plan.
- Multi-environment naming and coexistence.
- Platform bundle source instead of local filesystem source.

## 17. Why This Shape

The design keeps three responsibilities separate:

- `deploy-core` decides what the project needs in target-neutral terms;
- `deploy-dokploy` decides how Dokploy should materialize that plan;
- CLI/platform frontends decide where config/secrets come from and how deploys
  are verified.

That separation gives a useful Dokploy MVP without trapping future Fly.io,
Railway, or platform-managed deploy flows inside CLI-specific code. It also
keeps `@rntme/runtime` focused on one service process while the deployment
pipeline handles the project-level assembly around it.
